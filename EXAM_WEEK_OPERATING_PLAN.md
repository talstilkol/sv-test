# LumenPortal — Exam Week Operating Plan

> מטרה: להחזיק את הפורטל במצב שמשרת את המבחן הקרוב בלבד. כל החלטה נמדדת לפי השאלה: האם זה מקרב תלמיד ל-100 בקורס SVCollege AI & Full Stack?

## 1. כלל עליון

עד שהמבחן נגמר, לא פותחים פיצ'ר Priority 2 אם הוא לא סוגר אחד מאלה:

- כיסוי חסר בחומר SVCollege.
- טאב שלא עובד בדסקטופ או מובייל.
- שאלה שחוזרת על עצמה או לא מקדמת קושי.
- מושג בלי הגדרה קצרה, דרישות קדם או פידבק אחרי טעות.
- מצב שבו תלמיד מקבל 100 בלי לפתור שאלה קשה או קטע קוד.

## 2. סדר עבודה יומי

### בוקר — בדיקת כשירות

1. להריץ:

```bash
npm test -- --run
npm run build
npm run svcollege:readiness:release
npm run svcollege:tab-matrix:strict
npm run qa:questions:strict
npm run validate:strict
npm run exam:weakest:write
```

2. אם אחת הפקודות נכשלת, לא מוסיפים תוכן חדש עד שהיא ירוקה.
3. אם יש density warnings, מטפלים קודם במושגים שמופיעים במבחנים מדומים או במודולים החלשים ביותר.

### צהריים — סגירת פערי לימוד

1. לבחור 10 מושגים מתוך `EXAM_WEEK_WEAKEST_CONCEPTS.md`.
2. לכל מושג להוסיף או לאמת:
   - הגדרה בשורה אחת.
   - דרישות קדם.
   - שאלה קלה, בינונית וקשה.
   - שאלה אחת עם קוד אם המושג קוד-בייסד.
   - הסבר למה התשובה הנכונה נכונה.
   - הסבר למה כל distractor שגוי.
   - אסוציאציה קצרה לזיכרון.
3. לוודא שהמושג מתויג canonical כך שכל הטאבים מעדכנים אותו score bucket.

### ערב — סימולציית תלמיד

1. לפתוח פרופיל תלמיד מקומי נקי.
2. לעבור במסלול:
   - מושגים נטו קצר.
   - שיעור אחד מלא.
   - תרגול שאלות.
   - Code Trace או Bug Hunt.
   - מבחן מדומה קצר.
3. לוודא שאין שאלה חוזרת.
4. לוודא שתשובה נכונה מובילה לשאלה קשה יותר.
5. לוודא שטעות מוסיפה חולשה ומציגה הסבר קצר ומועיל.
6. לוודא ש-XP/status לא תופסים שטח לימוד מרכזי.

## 3. Gate לפני שחרור Exam Edition

Exam Edition מוכן רק אם:

- `svcollege:readiness:release` ירוק.
- `svcollege:tab-matrix:strict` ירוק.
- `qa:questions:strict` ירוק.
- `npm test -- --run` ירוק.
- `npm run build` ירוק.
- smoke ידני בדסקטופ עבר לכל הטאבים.
- smoke ידני במובייל עבר לכל הטאבים.
- אין קידום ל-100 בלי שאלת עומק והוכחת קוד למושגי קוד.

## 4. מה לא עושים השבוע

- לא בונים Teacher Dashboard.
- לא מרחיבים חנות או מוזיאון מעבר לתיקון באגים.
- לא פותחים פורטלים לקורסים אחרים.
- לא עושים migration גדול ל-TypeScript או Vite מעבר למה שכבר עובד.
- לא מוסיפים AI Tutor production לפני guardrails, rate limits ו-logging.

## 5. מדד הצלחה יומי

בסוף כל יום צריך להיות אפשר לומר:

- אילו 10 מושגים התחזקו היום.
- כמה שאלות עומק נוספו או אומתו.
- איזה טאב נבדק בפועל.
- כמה כפילויות שאלות נמנעו או נמצאו.
- מה הדבר האחד שמסכן כרגע ציון 100.

## 6. עדיפות תיקון באגים

1. Runtime error שמונע לימוד.
2. שאלה כפולה או score bucket כפול.
3. שאלה קשה בלי דרישות קדם.
4. הסבר ארוך/מעורפל שלא אומר "מה זה" בשורה אחת.
5. בעיית מובייל או פוקוס שמסתירה תוכן.
6. בעיית XP/coins/store שאינה משפיעה על לימוד חובה.
