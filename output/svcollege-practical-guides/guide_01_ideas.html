0. מפת רעיונות בלי כפילויות

מתוך דף המאסטר ללא כפילויות ומדריך הסבתא: הרבה נושאים במבחן הם אותו רעיון בלבוש אחר. אם מבינים את הקשר ביניהם, צריך לזכור פחות ולפתור מהר יותר.

אותו סיפור

Reference, מערכים ו־React State

Reference, שינוי מערך במקום, React immutable state ו־spread shallow copy הם אותו עיקרון: אם משנים את אותה קופסה, React עלול לא לזהות שינוי. יוצרים קופסה חדשה.

דימוי פשוט

האפליקציה כמו מסעדה

React הוא הדלפק שהמשתמש רואה. Express הוא הפקיד שמקבל בקשות. MongoDB הוא הארון שבו שומרים דפים. Mongoose הוא הטופס שמוודא שהדף כתוב נכון.

איך לחשוב במבחן

מסך → בקשה → שרת → מסד → מסך

כמעט כל פרויקט 70 נקודות חוזר על אותה זרימה: טופס ב־React, שליחה ב־fetch, route ב־Express, פעולה ב־Mongo, ואז עדכון state ורינדור מחדש.

0.5 דימויים קצרים שחוסכים בלבול

HTML / CSS / JavaScript

HTML הוא השלד, CSS הוא הצבע והריהוט, JavaScript הוא החשמל והכפתורים שגורמים למסך להגיב.

TypeScript

כמו מורה שבודק לפני ההגשה: אם אמרת שזה מספר ושלחת טקסט, הוא עוצר אותך בזמן כתיבת הקוד. הוא לא מחליף validation ל־JSON שמגיע מבחוץ.

useEffect

“אחרי שהמסך צויר, תעשה פעולה חיצונית”: fetch, timer או event listener. אם יוצרים listener או timer, צריך cleanup.

Router ו־Context

Router הוא שלטים בבית: כל כתובת מובילה לחדר. Context הוא מגפון להעברת מידע בלי להעביר אותו ידנית דרך כל קומפוננטה.

  • React לא מתחבר ישירות ל־MongoDB. הוא מדבר עם API.
  • Validation חשוב גם בצד הלקוח וגם בצד השרת.
  • Status codes הם חלק מהפתרון: 400 לקלט לא תקין, 404 לא נמצא, 500 תקלה.
  • State ב־React לא משנים עם mutation ישיר; יוצרים מערך או אובייקט חדש.
  • TypeScript לא מוכיח שנתונים חיצוניים תקינים בזמן ריצה; עדיין בודקים אותם.

שאלת פלט קוד

JS / Event Loop / Reference

שאל את עצמך: מה רץ סינכרונית, מה נכנס ל־Promise/microtask, מה מחכה ל־timer, והאם יש אובייקט או מערך שעובר by reference.

שאלת React

State / Props / Effects

בדוק: האם state השתנה ישירות, האם חסר key ברשימה, האם useEffect תלוי בערך שלא נמצא ב־deps, והאם צריך cleanup.

1. התמונה הגדולה: מה קורה כשמשתמש לוחץ?

Frontend

React

המסך: כפתורים, טפסים, רשימות, ניווט והצגת נתונים. React לא “שומר במסד”; הוא מנהל UI ומדבר עם API.

Backend

Node / Express

השרת: מקבל בקשות, בודק נתונים, מחזיר תשובות, ומדבר עם מסד הנתונים.

Database

Mongo / Mongoose

הזיכרון הקבוע: שומר רשומות. Mongoose מוסיף Schema/Model כדי לעבוד מסודר.

שלבמה קורהמה כותבים במבחן
1משתמש ממלא טופסuseState, controlled inputs, validation
2React שולח בקשהfetch עם method, headers, body
3Express מקבלapp.post, express.json(), status codes
4Mongo שומר/מחזירSchema, Model, find, save, update/delete
5React מרנדר מחדשstate חדש, map, empty/loading/error states

2. JavaScript Core: הבסיס של כל המבחן

משתנים

const כברירת מחדל. let כשמחליפים ערך. var ישן ומבלבל בגלל scope ו-hoisting.

Reference מול Value

Primitive מועתק כערך. Object/Array/Function מועתקים כהפניה. שינוי פנימי יכול להשפיע דרך כמה משתנים.

Async

סדר הרצה: קוד רגיל, microtasks של Promise/await, ואז macrotasks כמו timeout ואירועים.

console.log(1);
Promise.resolve().then(() => console.log(2));
setTimeout(() => console.log(3), 0);
console.log(4);
// סדר: 1, 4, 2, 3

3. React: לחשוב בקומפוננטות

State הוא זיכרון מקומי

כל input מחובר ל-state. כשמשנים state, React מרנדר מחדש. אם הערך הבא תלוי בערך הקודם, משתמשים ב-functional update.

setItems(prev => [...prev, newItem]);

לא עושים mutation ישיר

מערך/אובייקט ב-state הוא reference. שינוי במקום עלול לא להפעיל רינדור נכון. יוצרים reference חדש.

// נכון:
setForm(prev => ({ ...prev, title: value }));
Hookמתי משתמשיםנפילה נפוצה
useStateטפסים, רשימות, loading/error, toggleלעדכן אובייקט בלי spread
useEffectAPI, timer, event listener, משהו חיצוני ל-renderdeps לא נכונים או loop אינסופי
useReffocus, DOM ref, ערך שלא צריך לגרום renderלצפות ש-ref ירנדר מחדש
useMemoחישוב יקר או רשימה מסוננת/ממוינתלהשתמש במקום לפתור state פשוט

4. Routing, Forms, Validation: איפה רוב הנקודות בפרויקט

Routing

BrowserRouter, Routes, Route, Link, useNavigate, useParams. כל מסך צריך path ברור.

Forms

כל input עם value ו-onChange. אחרי submit: validate, fetch, update state, reset או navigate לפי הדרישה.

Validation

זה אזור ציון קריטי: שדות לא ריקים, סיסמה, מספרים, גיל, אורך טקסט, duplicate, הודעת שגיאה ברורה.

6. TypeScript ו־Next.js בקיצור

TypeScript

המטרה אינה “עוד קוד”, אלא חוזה: מה הצורה של המידע. למבחן חשובים type, interface, enum, union ו-type narrowing.

Next.js

React עם מבנה אפליקציה: App Router, תיקיות routes, layout משותף, dashboard/pages, Tailwind ו-deploy readiness.

7. מה לא ללמוד עכשיו

  • לא להחליף Stack שבוע לפני מבחן.
  • לא להיכנס ל-AI/DevOps/Nest לפני שסיימת Project 70, JS 20 ו-TS 10.
  • לא להשתמש בנתונים מומצאים. אם השאלה נותנת נתונים, משתמשים בהם; אם לא, המשתמש מזין.
  • לא להשתמש באקראיות Native כדי “לבחור” נתונים במבחן.
